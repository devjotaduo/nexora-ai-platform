# Nexora AI Platform 企业级工程治理与健壮性建设规范

> 更新时间：2026-05-30

本文档用于约束 Nexora AI Platform 后续开发，目标是确保"权限可控、审计可查、部署可复制、数据可迁移、升级可回滚、上游可同步"的工程标准。

## 1. 当前阶段

项目已完成从开源 QwenPaw 到企业级 AI 工作台的产品化改造：

- 登录认证、JWT 会话管理
- 用户管理、角色管理（admin / operator 双角色 RBAC）
- 三层权限：平台访问 → 智能体授权 → 能力审批
- 智能体授权（agent_grants）、批量授权/撤销
- 能力审批流（按风险等级、工具类型配置策略）
- 全链路审计日志（PostgreSQL 存储，分页检索）
- Token 消耗按用户/智能体/模型/日期统计
- 全部业务数据 PostgreSQL 持久化（Alembic 迁移管理）
- Docker 容器化（健康检查、资源限制、数据卷持久化）
- 单元测试 96 passed、100 用户并发压力测试通过
- 二开代码隔离在 `qwenpaw_ext/nexora` 和 `console/src/nexora`

平台已具备企业级试运行条件。

## 2. 开发红线

以下做法禁止进入主线，除非 PR 中明确标记为临时方案并给出清理日期：

- 写死用户名、密码、Token、API Key、模型 Key
- 只在前端隐藏菜单，不在后端校验权限
- 绕过 RBAC 直接调用工具、MCP、Skill
- 新增接口不记录审计
- 新增高危工具不接入审批策略
- 直接修改上游核心逻辑但不说明原因
- 使用 JSON 文件作为业务数据存储
- 生产部署依赖本机绝对路径
- 修改数据前不备份
- 为了赶进度跳过登录、权限、审计、Docker 验证

## 3. 二开代码组织原则

优先级从高到低：

1. 独立扩展模块：`src/qwenpaw_ext/nexora/`
2. 独立后端路由：`src/qwenpaw/app/routers/nexora.py`
3. 独立前端页面和 API 模块：`console/src/nexora/`
4. 少量接入点修改上游核心模块

必须避免：

- 在上游业务文件中堆大量 Nexora 专属逻辑
- 为了一个页面需求直接改通用组件行为
- 在多个地方复制同一套权限判断

推荐做法：

- 权限判断收敛到后端统一函数
- 菜单展示收敛到前端统一配置
- 工具授权收敛到智能体治理策略
- 审计写入收敛到统一审计函数
- Token 统计通过 ContextVar 自动传递当前用户

## 4. 权限体系规范

三层权限模型：

```text
第一层：用户 → 角色 → 菜单 / 接口权限
第二层：用户 → agent_grants → 智能体 → Tool / MCP / Skill
第三层：工具调用 → 审批策略检查 → 审批 / 直接执行
```

默认规则：

- 未登录用户不能访问受保护页面和 API
- 未授权角色不能看到对应菜单
- 未授权用户不能使用对应智能体
- 未授权智能体不能调用对应工具
- 高风险工具调用必须经过审批
- 后端是最终权限裁决点，前端隐藏只是体验优化

## 5. 数据存储规范

### 5.1 PostgreSQL 规范

- 所有业务数据必须存储在 PostgreSQL
- 表结构变更必须通过 Alembic 迁移脚本管理
- 每个迁移脚本必须包含 upgrade 和 downgrade
- 索引设计需考虑查询频率和数据量（审计日志按日期索引、Token 统计按日期+用户索引）
- 连接池配置需适配 100 用户并发

### 5.2 当前表结构

| 表名 | 用途 |
|------|------|
| nexora_users | 用户账号与角色 |
| nexora_audit_events | 审计日志 |
| nexora_approval_requests | 审批请求 |
| nexora_agent_grants | 用户-智能体授权 |
| nexora_agent_templates | 智能体模板 |
| nexora_governance | 资源治理策略 |
| nexora_capability_policies | 能力审批策略 |
| nexora_runtime_config | 运行时配置 |
| nexora_token_usage | Token 消耗记录 |

## 6. 审计规范

### 6.1 必须审计的事件

- 登录成功 / 失败
- 用户注册 / 删除 / 角色变更
- 智能体授权变更
- 审批请求创建 / 审批 / 拒绝
- 聊天消息发送 / 重连 / 停止
- 文件上传
- 配置变更
- 工具 / MCP / Skill 执行

### 6.2 审计字段要求

每条审计记录必须包含：

- 操作者（actor）
- 操作类型（action）
- 资源类型和 ID
- 操作结果（success / failure）
- 客户端 IP 和 User-Agent
- 操作详情（JSON，敏感参数脱敏）
- 时间戳

### 6.3 审计写入原则

- 审计写入失败不能阻断主流程
- 审计日志只追加，不修改不删除
- 后续需增加留存策略和归档机制

## 7. 安全规范

| 层级 | 要求 |
|------|------|
| 认证 | 所有受保护接口必须校验 JWT Token |
| 密码 | 使用 bcrypt 哈希存储，禁止明文 |
| RBAC | 新增接口必须声明所需角色权限 |
| 智能体访问 | 用户只能访问 agent_grants 中授权的智能体 |
| 工具安全 | Tool Guard 拦截危险命令，File Guard 限制敏感路径 |
| 审批 | 高风险工具调用必须经过审批流程 |
| Token 追踪 | Token 消耗必须关联到认证用户（通过 ContextVar） |
| 数据修改 | 修改密码/配置/权限前必须备份 |

## 8. 部署规范

### 8.1 环境要求

| 环境 | 要求 |
|------|------|
| 开发 | 本地源码启动，Docker 容器化 PG |
| 测试 | Docker Compose，独立数据卷，开启全部安全特性 |
| 生产 | 固定版本镜像，HTTPS + Nginx/网关，数据卷持久化，定期备份 |

### 8.2 资源建议

```text
应用容器：2 CPU / 4GB 内存
PostgreSQL：1 CPU / 1GB 内存
最大活跃智能体：20（可调）
空闲智能体回收：1 小时
```

### 8.3 生产禁止

- 不得直接暴露容器端口到公网
- 不得使用默认密码
- 不得关闭认证
- 不得使用 latest 标签部署生产环境

## 9. 版本发布规范

每次发布必须包含：

- Git 提交号和版本标签
- 变更摘要
- 数据迁移说明（如有表结构变更）
- 回滚方式
- 验证结果

镜像标签规范：

```text
ghcr.io/<org>/nexora-ai-platform:v0.1.0   # 生产
ghcr.io/<org>/nexora-ai-platform:latest    # 开发
```

## 10. 上游同步规范

- 二开优先放在扩展模块
- 修改上游核心文件必须记录原因
- 定期从 upstream 拉取更新，在隔离分支验证
- 每次同步必须跑完整验证清单

同步后验证清单：

- [ ] 登录 / 退出
- [ ] 用户权限管理
- [ ] 智能体授权
- [ ] 聊天功能
- [ ] 工具 / MCP / Skill 过滤
- [ ] 审计日志
- [ ] Token 消耗统计
- [ ] 审批流程
- [ ] 模型配置
- [ ] 前端构建
- [ ] 后端启动
- [ ] Docker 构建

## 11. 功能合入检查清单

新功能合入前必须回答：

- [ ] 是否复用了原项目已有能力？
- [ ] 是否只把二开逻辑放在必要位置？
- [ ] 是否影响上游升级？
- [ ] 是否需要 RBAC 权限？
- [ ] 是否需要智能体授权？
- [ ] 是否涉及工具执行？
- [ ] 是否需要审批？
- [ ] 是否记录了审计？
- [ ] 是否涉及密钥或敏感信息？
- [ ] 是否有测试？
- [ ] 是否影响 Docker 部署？
- [ ] 是否有表结构变更（需 Alembic 迁移）？
- [ ] 修改数据前是否备份？
