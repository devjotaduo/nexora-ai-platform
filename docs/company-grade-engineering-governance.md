# Nexora AI Platform公司级工程治理与健壮性建设规范

本文档用于约束Nexora AI Platform后续二次开发，目标是避免“临时可用”“轻量拼接”“只在本机跑通”的方案进入主线，为公司级、多用户、长期演进的智能运维平台打基础。

## 1. 当前阶段判断

当前项目已经完成从开源 QwenPaw 到Nexora AI Platform的初步产品化改造：

- 已具备登录认证、用户管理、角色管理、菜单权限、智能体权限、工具/MCP/Skill 权限、审计日志、中文品牌化、Docker 构建和部署文档。
- 已验证本地 Docker 镜像可以构建成功，容器可以启动并返回平台首页。
- 二开逻辑已经尽量放在 `qwenpaw_ext/nexora` 和前端独立页面/模块中，降低了与上游代码的耦合。

但从公司级平台角度看，当前仍处在“可用原型 + 初步工程化”阶段，不能直接视为大规模生产就绪。

## 2. 主要健壮性风险

### 2.1 存储层风险

当前用户、角色、治理策略、审计日志主要依赖本地文件存储：

- 权限策略：`nexora_governance.json`
- 审计日志：`nexora_audit.jsonl`
- 用户和角色：复用上游认证文件

这适合单机原型，但大量用户同时使用时会遇到：

- 并发写入冲突。
- 缺少事务能力。
- 查询和分页能力弱。
- 审计日志增长后读取性能下降。
- 数据结构演进缺少迁移机制。

公司级目标：

- 权限、用户、角色、审计、审批流必须迁移到数据库。
- 所有结构变更必须有版本化迁移。
- 审计日志必须支持分页、索引、归档和留存策略。

### 2.2 权限边界风险

当前权限体系已经形成两条主线：

- 用户 -> 角色 -> 菜单
- 用户 -> 角色 -> 智能体 -> 工具/MCP/Skill

但仍需继续强化：

- 后端必须作为最终权限裁决点，前端隐藏只能作为体验优化。
- 工具/MCP/Skill 的执行入口必须强制校验智能体授权，不能只在列表接口过滤。
- 默认策略必须保守，未授权即拒绝。
- 高危工具必须进入审批流，不能只靠角色授权。

公司级目标：

- 所有敏感 API 必须有权限声明。
- 所有工具执行必须记录审计。
- 所有高危动作必须支持审批、二次确认、执行人追踪和回滚说明。

### 2.3 审计能力风险

当前已实现基础审计，能够记录登录、用户管理、权限变更、工具/会话相关行为。

短板：

- 审计使用 JSONL 文件，不适合高频查询。
- 目前没有审计留存周期、归档、检索索引和导出机制。
- 对工具执行详情、输入输出脱敏、审批链路关联还不完整。

公司级目标：

- 审计日志进入数据库或专门日志系统。
- 每条审计必须包含：用户、角色、智能体、工具、资源、动作、结果、IP、User-Agent、请求 ID、审批 ID。
- 支持按时间、用户、智能体、工具、风险等级、状态检索。
- 对敏感参数和密钥做脱敏。

### 2.4 部署与运行风险

当前 Docker 镜像已能构建运行，但仍有公司级优化空间：

- 镜像体积约 4GB，偏大。
- 镜像依赖大量图形和浏览器运行组件，后续需要拆分运行模式。
- 当前 Compose 适合单机部署，尚未形成生产级高可用方案。
- 缺少健康检查、资源限制、日志轮转、备份恢复自动化。

公司级目标：

- 区分开发镜像、生产镜像、带浏览器能力镜像。
- 增加容器健康检查。
- 增加 CPU/内存资源建议。
- 数据目录、密钥目录、备份目录明确持久化。
- 形成标准升级、回滚、备份、恢复文档。

### 2.5 上游同步风险

本项目基于开源项目二次开发，上游还会持续更新。

风险：

- 修改上游核心文件越多，后续合并越难。
- 如果二开逻辑散落在上游模块中，升级会频繁冲突。
- 前端路由、菜单、认证中间件属于高冲突区。

公司级目标：

- 二开优先放在扩展模块。
- 修改上游核心文件必须记录原因。
- 定期从 upstream 拉取更新，并在隔离分支验证。
- 每次上游同步必须跑权限、登录、智能体、工具、审计、Docker 验证清单。

### 2.6 测试与质量风险

当前已有部分测试，但二开功能仍需要建立强制回归验证。

必须补齐：

- RBAC 单元测试。
- 智能体授权单元测试。
- 工具/MCP/Skill 授权测试。
- 登录、登出、菜单可见性、无权限访问的端到端测试。
- Docker 构建和容器启动验证。

公司级目标：

- 主分支不允许跳过关键测试。
- 每个权限变更必须附带测试。
- 每个部署变更必须附带本地或 CI 验证记录。

## 3. 后续开发红线

以下方案默认不得进入主线，除非在 PR 中明确标记为临时方案，并给出清理日期和替代方案。

- 写死用户名、密码、Token、API Key、模型 Key。
- 只在前端隐藏菜单，不在后端校验权限。
- 绕过 RBAC 直接调用工具、MCP、Skill。
- 新增接口不记录审计。
- 新增高危工具不接入审批设计。
- 直接修改上游核心逻辑但不说明原因。
- 使用本地临时文件作为长期数据存储。
- 生产部署依赖本机绝对路径。
- 部署文档只写“能跑”，不写升级、回滚、备份。
- 为了赶进度跳过登录、权限、审计、Docker 验证。

## 4. 二开代码组织原则

优先级从高到低：

1. 独立扩展模块：`src/qwenpaw_ext/nexora`
2. 独立后端路由：`src/qwenpaw/app/routers/nexora.py`
3. 独立前端页面和 API 模块。
4. 少量接入点修改上游核心模块。

必须避免：

- 在上游业务文件中堆大量Nexora 专属逻辑。
- 为了一个页面需求直接改通用组件行为。
- 在多个地方复制同一套权限判断。

推荐做法：

- 权限判断收敛到后端统一函数。
- 菜单展示收敛到前端统一配置。
- 工具授权收敛到智能体治理策略。
- 审计写入收敛到统一审计函数。

## 5. 权限体系目标模型

公司级权限模型固定为两条主线：

```text
用户 -> 角色 -> 菜单
用户 -> 角色 -> 智能体 -> 工具 / MCP / Skill / API / CLI
```

解释：

- 菜单权限决定用户能看到和进入哪些页面。
- 智能体权限决定用户能看到、使用、管理哪些智能体。
- 工具权限挂在智能体上，用户不能直接越过智能体调用工具。
- 高危工具在“智能体可调用”之外，还要进入审批流。

默认规则：

- 未登录用户不能访问受保护页面和 API。
- 未授权角色不能看到对应菜单。
- 未授权角色不能使用对应智能体。
- 未授权智能体不能看到或调用对应工具。
- 未配置授权策略的生产资源默认不可用。

## 6. 数据层演进路线

### 阶段 1：当前文件存储稳定化

- 给权限文件和审计文件增加版本号。
- 增加读写异常告警。
- 增加写入前备份。
- 增加审计日志轮转。

### 阶段 2：引入 SQLite

适合单机生产或小规模团队：

- 用户、角色、权限、智能体授权、工具授权、审计日志进入 SQLite。
- 增加 Alembic 或同等迁移机制。
- 支持备份和恢复。

### 阶段 3：引入 PostgreSQL

适合公司级多人使用：

- 所有核心业务数据进入 PostgreSQL。
- 审计日志表按时间索引。
- 支持多实例共享数据。
- 后续可接 Redis 做缓存和队列。

## 7. 审批流建设原则

审批流不是简单页面，而是工具执行前的安全闸口。

必须包含：

- 申请人。
- 使用的智能体。
- 目标工具/MCP/Skill。
- 操作参数摘要。
- 风险等级。
- 审批人。
- 审批结果。
- 执行结果。
- 审计关联 ID。

执行原则：

- 低风险动作可直接执行但必须审计。
- 中风险动作可配置是否审批。
- 高风险和关键动作必须审批。
- 审批通过后只能执行当次申请，不能长期放权。

## 8. 部署形态规划

### 开发环境

- 本地源码启动。
- Docker 本地构建验证。
- 允许使用临时端口，但不得写入生产文档作为默认方案。

### 测试环境

- Docker Compose 部署。
- 独立数据卷。
- 独立测试模型 Key。
- 开启认证、权限、审计。

### 生产环境

- 使用固定版本镜像。
- 使用 Nginx 或网关暴露 HTTPS。
- 数据库和密钥目录持久化。
- 配置备份、日志轮转、监控告警。
- 不允许直接暴露未保护的容器端口到公网。

## 9. 版本与发布规范

每次发布必须包含：

- Git 提交号。
- 版本号或镜像 tag。
- 变更摘要。
- 数据迁移说明。
- 回滚方式。
- 验证结果。

镜像 tag 建议：

```text
ghcr.io/your-org/nexora-ai-platform:v0.1.0
ghcr.io/your-org/nexora-ai-platform:latest
```

生产环境优先使用固定版本号，不直接依赖 `latest`。

## 10. 当前整改优先级

### P0：近期必须补齐

- 增加公司级 PR 检查清单。
- 增加 RBAC 和智能体授权测试。
- 增加 Docker Compose 健康检查。
- 明确生产部署禁止直接公网暴露。
- 梳理所有工具执行入口，确认后端权限强制校验。

### P1：进入试点前完成

- 审计日志分页、导出、轮转。
- 审批流第一版。
- 权限数据版本号和迁移脚本。
- Docker 镜像瘦身和生产镜像拆分。
- CI 自动构建和基础测试。

### P2：公司级推广前完成

- 数据库化。
- PostgreSQL 部署模式。
- 多实例运行设计。
- SSO/LDAP/OIDC 接入评估。
- 操作审计留存策略。
- 高危工具审批策略模板。

## 11. 每次新增功能的检查清单

新增功能合入前必须回答：

- 是否复用了原项目已有能力，避免重复建设？
- 是否只把二开逻辑放在必要位置？
- 是否影响上游升级？
- 是否需要菜单权限？
- 是否需要智能体权限？
- 是否涉及工具/MCP/Skill 执行？
- 是否需要审批？
- 是否需要审计？
- 是否涉及密钥或敏感信息？
- 是否有测试？
- 是否影响 Docker 部署？
- 是否有升级和回滚说明？

## 12. 结论

当前项目已经具备公司级平台雏形，但还没有达到大量用户生产使用的健壮性要求。

后续开发必须从“功能能跑”切换到“权限可控、审计可查、部署可复制、数据可迁移、升级可回滚、上游可同步”的工程标准。
