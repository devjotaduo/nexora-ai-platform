## 三、100 用户并发压力测试

### 3.1 测试方案

**工具**: Locust 2.34.0  
**并发用户数**: 100  
**拉起速率**: 10 用户/秒  
**持续时间**: 60 秒  
**用户分布**: ~10% admin (admin) + ~90% operator (9 个业务用户)  

**测试场景（11 类操作，权重分配）**：

| 操作 | 权重 | 适用角色 | 说明 |
|------|------|----------|------|
| 智能体列表 | 5 | 所有 | `GET /api/agents` |
| Skill 列表 | 5 | 所有 | `GET /api/skills` |
| 发送聊天 | 4 | 所有 | `POST /api/agents/{id}/chats` |
| 聊天历史 | 3 | 所有 | `GET /api/agents/{id}/chats` |
| 审计日志 | 3 | admin | `GET /api/nexora/audit/events` |
| 审批列表 | 2 | admin | `GET /api/nexora/approval-requests` |
| 治理策略 | 2 | admin | `GET /api/nexora/governance/policies` |
| 用户管理 | 2 | admin | `GET /api/auth/users` |
| 智能体授权 | 1 | admin | `GET /api/nexora/agent-grants/{id}` |
| SPA 页面 | 1 | 所有 | 随机访问前端页面 |
| 静态资源 | 1 | 所有 | `GET /logo-icon.svg` |

**关键设计**：
- operator 用户仅访问已授权的智能体，避免产生非业务错误
- admin 用户独享管理类接口（审计/审批/治理/用户管理）
- 聊天消息携带 session_id + user_id + message，模拟真实交互

### 3.2 测试结果

#### 核心指标

| 指标 | 数值 | 评估 |
|------|------|------|
| 总请求数 | 1,640 | — |
| 错误率 | **0.37%**（6 次） | 优秀 |
| 吞吐量（QPS） | 27.4 req/s | 满足 100 人需求 |
| 平均响应时间 | 640ms | 良好 |
| P50 响应时间 | 470ms | 良好 |
| P95 响应时间 | 1,800ms | 可接受 |
| P99 响应时间 | 2,400ms | 可接受 |
| 最大响应时间 | 2,827ms | 无超时 |

> **P95**：95% 的请求在此时间内完成  
> **QPS**：每秒处理的请求数，衡量系统吞吐能力

#### 错误分析

全部 6 次错误均为 TCP `ConnectionResetError(54, 'Connection reset by peer')`，分布在 4 个端点，为单 worker 高并发下的正常现象，无任何业务逻辑错误（无 4xx/5xx 业务错误）。

| 发送聊天 | 320 | 1 | 782 | 660 | 2,100 | 2,400 |

| 审批列表 | 15 | 0 | 549 | 310 | 1,600 | 1,600 |
| 治理策略 | 11 | 0 | 436 | 220 | 1,800 | 1,800 |
| 用户管理 | 5 | 0 | 503 | 490 | 1,000 | 1,000 |

### 3.3 压力测试结论

- **业务错误率 0%** — 全部错误为 TCP 连接重置，非业务缺陷
- **RBAC 权限隔离有效** — operator 用户全程无越权访问
- **PG 连接池稳定** — 无连接超时或连接池耗尽
- **100 人规模完全满足** — 单 worker 下 27 QPS，人均每 4 秒一次请求

## 五、遗留事项与优化建议

### 非阻塞遗留项

| 事项 | 优先级 | 说明 |
|------|--------|------|
| 聊天页智能体选择体验优化 | 低 | 当前切换智能体体验可进一步改善 |
| JWT secret 迁移 PG 存储 | 中 | 当前 JWT secret 在内存生成，重启后所有 Token 失效 |

### 性能优化建议

| 建议 | 预期效果 |
|------|----------|
| 生产部署 2-4 个 uvicorn worker | P95 可压到 500ms 以内（需改造 MultiAgentManager 为跨进程共享） |
| 前置 Nginx 做静态资源缓存 | 减轻 Python 进程负担，静态资源响应降到 10ms 以下 |
| PG 连接池按 worker 数调整 | 多 worker 时 pool_size 需按 worker 数等比缩小 |
